## 二、个人工作详述

### 2.1 4 路召回策略设计（贡献占比：100%）

**问题背景**：面对 137 万用户 × 10.5 万商品的候选空间，直接在全部商品上做排序计算不可行（时间复杂度 O(N·M) 过高）。行业通行做法是先通过高效的召回策略将候选集压缩至每用户数十到数百个候选，再做精排序。

**我的设计**：

我设计了 **4 路互补召回策略**，每路覆盖不同的推荐信号源：

| 召回路 | 召回逻辑 | 权重 | 数学描述 |
|:---|:---|---:|:---|
| **复购召回** | 按 `10⁹ × day_rank + volume_rank` 综合排序（时间优先、购买频次次优先） | 40% | `score(u,i) = α·day_rank(u,i) + β·volume_rank(u,i)` |
| **Item2Item** | 基于用户已购商品的共现频率，推荐最常搭配的同类商品 | 30% | `score(u,i) = Σ_{j∈Bought(u)} cooccur(i,j)` |
| **品类热门** | 用户已购买品类内，按销售热度排序 | 20% | `score(u,i) = sales_rank(i) · I(cat(i) ∈ Cat(u))` |
| **全局热门** | 全平台最热商品兜底（保障冷启动用户基础推荐） | 10% | `score(u,i) = global_sales_rank(i)` |

**关键技术决策**：

1. **复购召回排序函数**：我设计了 `10⁹ × day_rank + volume_rank` 的复合排序表达式。其中 `10⁹` 的系数确保时间维度优先级远高于购买频次——这是对"时尚商品时效性极强"这一业务认知的量化表达。在代码实现中，这是一个简单但有效的词法排序（Lexicographic Ordering），它比加权乘积更稳定，不会因数值范围差异导致某一维度被忽略。

2. **重叠处理**：不同召回路线的候选集存在重叠（最热门的商品可能同时出现在全局热门和品类热门中）。我实现了基于 `drop_common_user_item` 的去重逻辑——按召回优先级降序处理，低优先级的候选中如果出现已存在于高优先级候选集中的 user-item 对，则丢弃。这保证了候选集的多样性，同时避免了重复计算。

3. **候选截断**：合并所有召回路线后，我按融合分数排序截断至每用户 20 个候选。20 是一个经验值——太大会增加排序阶段的计算开销，太小会降低召回覆盖率。验证集实验表明，20 个候选可以覆盖约 95% 的 ground-truth 商品（Recall@20 ≈ 0.95）。

**量化成果**：4 路召回使 Baseline 热门推荐的 MAP@12 从 **0.010 提升至 0.018**（+80%），这是整个项目中最关键的性能跃升。

### 2.2 LightFM SVD 嵌入模型（贡献占比：100%）

**问题背景**：用户-商品交互矩阵极度稀疏（稀疏度 > 99.99%），直接使用 ID 特征进行 One-hot 编码会导致维度灾难。需要一种方式将高维稀疏信号压缩为稠密表示。

**我的方案**：

我选择 **LightFM 框架** 而非自行实现 SVD 分解，原因有三：
1. LightFM 原生支持 BPR Loss（Bayesian Personalized Ranking），这是隐式反馈场景下的最优损失函数之一
2. 它支持用户/商品的辅助特征融合（可在后续版本中扩展）
3. 底层使用 Cython 优化，训练速度远快于纯 Python 实现

**训练配置**：

| 参数 | 取值 | 选择理由 |
|:---|:---|:---|
| 损失函数 | BPR | 隐式反馈（购买/未购买）而非显式评分，BPR 通过最大化正样本与负样本的得分差来学习 |
| 嵌入维度 | 64 → 16（用于输入排序模型） | 64 维保留完整信号，16 维用于特征输入以控制过拟合 |
| 学习率 | 0.005 | 经过网格搜索验证，高于 0.01 训练不稳定，低于 0.001 收敛过慢 |
| 优化器 | Adadelta | 自适应学习率，无需手动调节衰减率 |
| Epochs | 100 | 早停观察约 50 轮后损失趋于稳定，100 轮确保充分收敛 |

**多时间窗口训练**：我特别设计了**多时间窗口嵌入策略**——分别在距离预测时间点 1~13 周的用户-商品交互矩阵上训练独立的 LightFM 模型。这意味着每个模型捕捉的是不同时间尺度下的用户偏好信号：
- 近 1-2 周的嵌入反映**即时冲动购买**模式
- 近 4-6 周的嵌入反映**月度穿搭习惯**
- 近 8-13 周的嵌入反映**季节偏好**

这一设计在后续的特征工程中通过周对齐（用当前周对应的嵌入）输入排序模型，使排序模型能够感知用户偏好的时变特性。

**量化成果**：加入 LightFM 嵌入特征后，MAP@12 从 **0.018 提升至 0.021**（+0.003），是召回之后贡献最大的单步提升。消融实验（移除嵌入特征）证实了其不可或缺性。

### 2.3 双模型融合排序与调参（贡献占比：100%）

**问题背景**：单模型存在偏置和方差，且在排序任务中，不同损失函数对相同的候选集会给出不同的排序顺序。融合可以平滑单模型的波动。

**我的实现**：

我分别使用 **LightGBM** 和 **CatBoost** 在相同的数据上进行训练，然后将两者的预测概率进行简单平均融合。

**LightGBM 配置**：

| 参数 | 取值 | 理由 |
|:---|:---|:---|
| objective | `xendcg` | 直接优化 NDCG/CG 指标，与 MAP@12 的目标最为接近 |
| boosting_type | `gbdt` | 传统梯度提升树，对表格数据最稳定 |
| learning_rate | 1e-6 | 极低学习率配合早停，防止过拟合 |
| num_leaves | 255 | 控制树复杂度，255 是经验合理值 |
| min_data_in_leaf | 100 | 防止叶节点只学到极少数样本的模式 |
| metric | `map` + `eval_at=12` | 直接以 MAP@12 作为验证指标，与竞赛评分完全对齐 |
| early_stopping | 20 轮 | 验证集 20 轮不提升即停止，防止过拟合 |

**CatBoost 配置**：

| 参数 | 取值 | 理由 |
|:---|:---|:---|
| loss_function | `YetiRank` | CatBoost 的 pairwise 排序损失，与 LightGBM 形成互补 |
| one_hot_max_size | 300 | 高基数类别特征自动处理，无需预编码 |
| iterations | 10000 | 配合早停，给模型足够收敛空间 |
| use_best_model | True | 自动选择验证集最优迭代点 |

**融合策略**：

我尝试了两种融合策略：
1. **Rank Fusion**：按两个模型对同一 user-item 对的排序位置求和，再排序
2. **Probability Average Fusion**：直接对两个模型的预测概率求平均，再排序

验证集实验表明，概率平均融合优于 Rank Fusion（+0.0003 MAP@12），且实现更简单。最终提交采用 LightGBM : CatBoost = **0.5 : 0.5** 的等权平均。

**3 折分组验证**：考虑到排序任务中同一个用户的数据不能被拆分到不同折中（会导致数据泄漏），我实现了按 user 分组的 3 折验证。每折中同一个用户的所有样本必须在同一折内，且验证集分布与训练集保持时间一致性。融合模型的 3 折平均 MAP@12 方差仅为单模型的 60%，说明融合显著提升了稳定性。

**量化成果**：双模型融合相比最优单模型（LightGBM）提升 **+0.001 MAP@12**，且 3 折方差降低 **40%**。

## 三、关键技术难点与解决过程

### 难点 1：候选集生成的 O(N·M) 复杂度问题

**问题**：如果对全量 137 万用户 × 10.5 万商品做笛卡尔积生成候选，中间结果将达到万亿级别，远超内存容量。

**我的解决思路**：观察到推荐系统的核心规律——大多数用户只与极少数商品有过交互（中位数 4 条记录），因此候选集只需要在"用户可能感兴趣"的局部空间中生成。我通过 4 路召回各自独立计算候选（每路都是 O(N) 或 O(K·logK)），再合并去重。由于每路都利用了数据本身的稀疏性（复购只发生在已购商品上，热门只需取 Top-K），实际计算量从 O(N·M) 降为 O(N·K)，其中 K << M（K=20, M=100k+）。

### 难点 2：BPR 损失训练时负样本的选择

**问题**：BPR 的训练需要为每个正样本（已购买）采样负样本（未购买）。简单的随机采样会选中大量用户从未接触过的商品，导致"易区分但无信息量"的训练信号。

**我的解决思路**：LightFM 默认的负采样策略是在全量商品中均匀随机采样。我分析了验证集的错误类型，发现模型的主要错误是"将完全无关的商品排在了应推荐商品之前"。为此我实现了**难负样本采样策略**——优先采样那些在品类热度上与正样本接近但未被购买的商品作为负样本。这使模型需要学习更精细的决策边界。但该实现增加了约 30% 的训练时间，且 MAP@12 仅提升 0.0004，考虑到性价比，最终版本中降级为 LightFM 默认负采样 + 更多 Epochs 的替代方案。

### 难点 3：LightGBM xendcg 的训练不稳定性

**问题**：使用 `xendcg` 目标函数时，我发现训练曲线在前 100 轮剧烈波动，验证集 MAP 忽高忽低，早停难以稳定触发。

**我的解决思路**：查阅 LightGBM 文档后发现，`xendcg` 在 `learning_rate` 过高时容易出现训练震荡。我将学习率从默认的 0.1 逐步下降至 **1e-6**，配合 `early_stopping` 使用。极低学习率下模型虽然收敛慢，但训练曲线极度稳定，且最终的泛化性能优于高学习率版本。这是一个直观上违反直觉但实践中有效的决策——在排序任务中，稳健优化比快速收敛更重要。

## 四、个人收获与能力成长

### 4.1 技术能力收获

1. **工业级推荐系统架构能力** — 我完整实践了"召回 + 排序"的经典推荐系统架构。以前我理解的推荐就是"一个模型搞定一切"，但实际项目中，召回和排序的分工完全是解决不同性质的问题——召回解决"从哪找"，排序解决"选哪个"。这种架构思维对我理解工业级系统的设计有深远影响。

2. **矩阵分解与隐式反馈建模** — 使用 BPR Loss 训练用户-商品嵌入的过程让我真正理解了"隐式反馈 ≠ 显式评分"这一关键区别。显式评分场景（如豆瓣电影评分）可以用 MSE 等回归损失，但隐式反馈场景（如电商购买）只有"正样本"（已购买）和"未标注"（未购买）两种信号，需要用 BPR 这样的 pairwise ranking loss 来建模。这一经验可以推广到任何隐式反馈场景（推荐、搜索、广告）。

3. **排序模型的工程调优** — LightGBM 的 `xendcg` + 极低学习率的组合、CatBoost 的 YetiRank + 自动类别特征处理、概率平均融合 vs Rank Fusion 的选择——这些不是课堂上能学到的知识，而是在反复实验和结果对比中摸索出的工程经验。

4. **Kaggle 竞赛方法论** — 我深刻理解了时序验证在竞赛中的关键地位。早期模型在随机切分上 MAP@12 ≈ 0.06，而严格时序切分下仅 ≈ 0.02，相差 3 倍。如果不做时序验证，我们可能会带着"0.06 分"的幻觉去提交，最终在 Private LB 上被 Shake-down。

### 4.2 思维方式的转变

- **从"堆模型"到"做减法"**：项目初期我倾向于堆砌更多模型（甚至考虑过 Transformer），但实际效果远不如在召回和特征工程上做深。我学会了在遇到性能瓶颈时，先溯源归因而非盲目上复杂模型。
- **特征的重要性不等于预测的贡献**：树模型的特征重要性排名具有误导性——年龄特征在 LightGBM 中排名前 10，但消融移除后 MAP 反而上升。从此我不再信任任何不经消融验证的特征重要性输出。
- **极致内存管理是数据工程的基础素养**：处理 3178 万条数据时，数据类型压缩、分块计算、稀疏矩阵存储不再是"锦上添花"而是"生存必需"。这一教训将直接影响我未来的数据处理习惯。

### 4.3 团队协作

我与祝帅的分工明确且互补——我专注于算法架构和模型训练，他负责数据清洗、特征工程和内存优化。合作中我学到的最重要的一点是：**在数据科学项目中，数据处理和特征工程往往决定了性能的上限，而模型选择只是在逼近这个上限**。最终模型从 0.022 到 0.023 的 +0.001 提升，远小于特征工程从 0.010 到 0.022 的 +0.012 提升——这个量级对比本身就是一个有力的结论。

## 五、自评与反思

### 5.1 做得好的

- **架构设计合理**：4 路召回 + SVD 嵌入 + 双模型融合的架构在工程上稳定、可复现、可解释，在 Top 4% 的排名上验证了其有效性
- **业务理解深入**：从"快时尚场景下的短期复购优先"这一洞察出发，设计了针对性的召回权重和特征体系，而非盲目套用通用推荐方案
- **实验严谨**：所有决策都有消融实验支撑，所有分数都基于严格时序验证，杜绝了数据泄漏导致的结果虚高

### 5.2 可以做得更好的

- **多模态信息未利用**：项目中完全没有使用商品图片和文本描述，而这是提升冷启动效果最可行的路径。如果再多一周时间，我优先想尝试的是用预训练 CNN（ResNet50）提取商品图片嵌入作为额外的特征
- **召回多样性可以更高**：4 路召回倾向于推荐"安全"商品（复购 + 热门），对长尾新颖商品的覆盖不足。引入 GNN 或 GraphSage 进行高阶协同过滤是我下一步想尝试的方向
- **调参效率低**：LightGBM 的学习率从 0.1 降到 1e-6 花了多轮实验。如果能更早使用 Optuna 或 Hyperopt 等自动调参工具，可以节省约 30% 的调参时间

在 H&M 个性化时尚推荐项目中，我负责了召回策略、嵌入模型和排序融合三大核心算法模块的设计与实现。最终方案在 Kaggle 竞赛中获得 **116/2952（Top 4%）** 的成绩，验证了"召回 + 排序 + 矩阵分解"技术路线在 3000 万级数据、百万级用户、十万级商品的大规模推荐场景下的有效性。
